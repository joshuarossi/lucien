§00 Methodology

A fine-tune is only as trustworthy as the fairness of its test. This evaluation is built so that the adapter is the only moving part — same conversations, same system prompt, same user prompt, same decoding parameters. Everything below is reproducible from benchmark/results/chunk-eval/.

The parity contract

Every model receives the exact input the chunk adapter was trained on (see make-finetune-data.ts): a fixed system prompt plus a user message of CHUNK_PROMPT (with the Meta policy inlined) followed by the formatted conversation. No coding-assistant system prompt, no <<LUCIEN_INTERNAL>> sentinel, no repository context files, no tools.

To prove the inputs are identical, every call records a SHA-256 over system + " " + user. The report asserts these hashes match across gold, base, and adapter for all 25 conversations:

✓ per-conversation prompt hash — e.g. conv #1 = — identical across all models

The frozen 25-conversation set is a held-out benchmark, explicitly excluded from the adapter's training data, so the adapter is being tested on unseen conversations.

✓ leakage audit — 0 of 25 eval conversations found across 1455 training examples — eval set is unseen

How each model was run

  1. Gold / benchmark — production pipeline: the chunks the live Lucien pipeline already wrote to lucien.db for these conversations. The reference is the system's actual running behavior, not an external model's preference.
  2. Base CPM — MiniCPM5-1B-OptiQ, served by mlx-bun, called with no adapter.
  3. Adapter CPM — the same base weights with the fine-tuned LoRA mounted at id "chunk", selected per-request.
  4. Judge — Opus 4.8: blind-scores each contestant's segmentation against the GPT-5.5 gold (boundary + label, 0–100, with a verdict). A third model family, and not a contestant, so there is no self-grading bias.

Local models are called by direct HTTP to the mlx-bun server (pi cannot carry the per-request adapter field); the cloud gold uses the same explicit system+user via pi with context-files and tools disabled.

The two local requests, side by side — the adapter field is the entire difference:

curl http://localhost:8090/v1/chat/completions \ -H 'content-type: application/json' \ -d '{ "model": "local", "messages": [ { "role": "system", "content": <chunk system prompt, verbatim, below> }, { "role": "user", "content": <CHUNK_PROMPT + Meta policy + formatted conversation> } ], "temperature": 0, "max_tokens": 8192, "stream": false }' # BASE — no adapter field
curl http://localhost:8090/v1/chat/completions \ -H 'content-type: application/json' \ -d '{ "model": "local", "adapter": "chunk", ◄── THE ONLY DIFFERENCE "messages": [ { "role": "system", "content": <identical> }, { "role": "user", "content": <identical> } ], "temperature": 0, "max_tokens": 8192, "stream": false }' # ADAPTER — fine-tuned LoRA mounted at id "chunk"

System prompt sent to all three models, verbatim (sysprompt-v1.json → stages.chunk.systemPrompt):

You are a precise conversation-segmentation engine, not a coding assistant and not a chat assistant. You receive ONE conversation transcript and return its topic segmentation. Absolute output contract (overrides any conversational instinct): - Your entire response MUST be a single JSON object, parseable by JSON.parse with no edits. - No prose, no preamble, no explanation, no follow-up question, no markdown code fences, no tool calls. The first character you emit is "{" and the last is "}". - Never ask for clarification or more data — the transcript in the message IS the data. If it has no meaningful content, return {"chunks": []}. - Use only message UUIDs present in the provided transcript; never invent or alter a UUID. Follow the segmentation rules and JSON schema in the message exactly.
