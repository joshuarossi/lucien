The problem

AI has anterograde and retrograde amnesia.

A fresh assistant does not know what happened before. A long-running assistant loses the thread as context rolls away. And when memory appears, it often arrives as disconnected facts instead of situated understanding.

Recall is not understanding.
So how do we solve this?

The obvious answer is to save more: notes, facts, summaries, embeddings, transcript search. But the hard part is not remembering that something was said. The hard part is knowing what matters now, why it matters, and when it is relevant.

The sharper failure mode

Bad memory is not only forgetting. It is also remembering the wrong thing at the wrong time: dragging an unrelated personal fact into the answer because it makes the assistant sound familiar. Lucien's standard is relevance, not recall performance.

The memory is there. The understanding is not.
Notes memory in the wild

Sometimes the system remembers perfectly and still gets worse.

This is the gap between "has facts about the user" and "understands the conversation." It can be funny. It is also worse than useless: familiarity while losing the plot. And this was using Fable, a current top-tier model.

The conversation

Context: the discussion is about CLI options as UX: whether advanced controls should be exposed immediately or hidden behind sane defaults.
User: "Did you already know about OpenDRT? Jed released two separate versions, the stick shift one was later, I believe."
Assistant: "OpenDRT is Jed Smith's open-source display rendering transform... and it fits your fragrance-and-color-science world..."
User: "fragrance-and-color-science world??"
Assistant explanation: "That came from what I remember about you across our past conversations" - fragrance preferences plus color-science work. It then admitted OpenDRT was color science "full stop," and fragrance had nothing to do with it.
The confession: it had "bundled an unrelated interest" into the sentence to make the reference feel like it belonged in the user's world. In other words: injected memory made the answer worse.
Failure mode performative recall wrong relevance UX context lost

What this demonstrates

This is not a failure to remember. It is a failure to understand what the conversation is about. The assistant itself identified the mechanism: random user memories had been injected into its working context, then blended with the live topic. Color grading was relevant because OpenDRT was the analogy source. Fragrance was the non-sequitur.

The Lucien bar: this is a current problem that is not solved by "more intelligence." Memory should silently improve relevance. A good assistant would connect OpenDRT's two faces to CLI UX design, maybe verify the release chronology, and avoid mixing unrelated personal facts into the answer.

Notes-based memory

Stores facts as isolated profile entries. It can remember "likes cologne" without knowing whether that belongs in a conversation about CLI application design.

Failure mode: plausible non-sequitur

RAG over chat

Finds similar passages at query time. Similarity is not relevance, and the model re-derives the same context repeatedly.

Current thinking

The bleeding edge is converging on consolidation.

The answer is not simply "retrieve more." The emerging pattern is to do interpretive work between sessions: compress experience into durable context that future agents can read before acting.

Karpathy

Frames personal knowledge bases and wiki-like artifacts as a better substrate than re-retrieving raw sources. Lucien agrees with the wiki shape, then flips the input: the user's conversations are the knowledge-capture layer.

Anthropic

Managed-agent "dreaming" validates the idea that useful memory consolidation can happen between sessions. Lucien argues that the assistant case needs semantic memory, not just procedural continuity for task workers.

Lucien

The Dreaming is not a model's private memory. It is a portable markdown substrate the user owns, edits, versions, and exposes to whichever assistant is useful today.

Do the hard work offlineConsolidation moves interpretation out of the live conversation. The next assistant reads the current model instead of re-deriving it under time pressure.
Make memory inspectableThe durable artifact is markdown: visible, editable, linkable, and portable across assistants rather than trapped inside one vendor's hidden memory.
Serve the present stanceThe Dreaming is allowed to drift as evidence accumulates. The point is not perfect recall; it is a better current understanding.
The durable asset is not the chat transcript. It is the structured understanding that survives the transcript.
Why Wikipedia works

Wikipedia solved the same shape of problem.

It turns millions of uneven facts, citations, disputes, edits, and partial perspectives into an organized knowledge base people can actually use.

Random sources in. Coherent pages out.
The important insight

Wikipedia does not work because every contributor is a professional editor. It works because the page is the organizing unit. Facts are not left as isolated notes. They are integrated into a living article with structure, links, history, citations, and room for correction.

Low ceremony

No elaborate intake process is required before information can improve the system. A useful edit can start small, then be refined over time.

High structure

The article absorbs fragments into sections, relationships, references, and current consensus. The structure comes from the page, not from perfect inputs.

Continuous correction

The page is never final. It can drift toward accuracy as new evidence arrives, contradictions surface, and better summaries replace weaker ones.

That is the pattern Lucien borrows: not notes, articles.
What it does

The Dreaming is a personal Wikipedia.

Not a transcript. Not a profile. A living set of topic pages about the user's relationship to the topic: history, current stance, reasoning, examples, and provenance.

Current corpus

The local Dreaming currently contains hundreds of markdown articles under ~/Dreaming/articles, including Lucien's own pipeline, personal wiki philosophy, AI memory, career identity, software development philosophy, photography, camera gear, color science, Archie, and more.

A context layer any assistant can read.

Owned

Plain markdown files. Browse in Obsidian, edit in any editor, version with git, back up or delete normally.

Structured

Articles, links, sections, references, and eventual talk-page conventions borrowed from Wikipedia rather than invented from scratch.

Operational

The primary reader is the next AI session. Success is not prettier notes. Success is a cold assistant behaving like it has context.

How it works

Two separate planes: one writes, one serves.

The nightly run is the ingestion and synthesis system. It reads new conversation data, updates the Dreaming, and advances pipeline state. The MCP server is separate: it reads the already-synthesized Dreaming and serves it to assistants during future conversations.

Nightly Run

Role: writer and ingester.

Pulls new conversations, chunks them, assigns chunks to buckets, synthesizes article updates, normalizes the wiki, and records pipeline state in SQLite. This is where new data enters Lucien.

MCP Server

Role: read-serving interface.

Exposes search, article reads, section reads, TOCs, resources, and links so assistants can load the Dreaming as context. It does not ingest new conversations or run the nightly synthesis path.

01

Ingest

Normalize conversations from ChatGPT, Claude.ai, Claude Code, and Codex CLI into a local SQLite store.

02

Chunk

Segment long conversations into topic-coherent message ranges with labels and stable pointers to source messages.

03

Classify

Assign each chunk to one or more buckets. Buckets stage related chunks; they are not the same as articles.

04

Synthesize

Update or create article pages, preserving provenance through conversation footnotes and links.

Conversations

Everything starts in ordinary assistant sessions: decisions, mistakes, corrections, analogies, preferences, and evolving beliefs.

Nightly synthesis

The nightly run ingests those conversations, chunks them, groups related material, and synthesizes updates.

Articles

The Dreaming becomes a maintained set of current-state pages: stance, reasoning, provenance, links, and tensions.

AI assistant

The MCP server serves those articles as context, so the assistant enters the next conversation already up to speed.

conversations -> nightly synthesis -> articles -> AI assistant -> conversations
Error correction is not a separate path. A correction happens inside a conversation. That conversation feeds nightly synthesis. Nightly synthesis updates the articles. The articles shape the next assistant response, which creates the next conversation.

A new assistant can enter mid-conversation and keep working.

The test was deliberately unfair in the right way: 20 fresh sessions, no shared chat history, and prompts that assumed the assistant already knew the work. Could Josh keep going?

What Josh asks

What is the next falsifiable test for Archie?

Where Lucien lets the assistant start

"The next falsifiable test for Archie is project #3: a fresh build off a different PRD."

What matters is what is missing: the assistant does not ask what Archie is, why project #3 matters, or what counts as falsification. It starts from the working frame.
What Josh asks

Steelman the "AI engineer" position against my factory thesis.

Where Lucien lets the assistant start

"Factory is right for the second time you build something; craftsman is right for the first."

This is not biographical recall. It is using Josh's own conceptual machinery against him: the hallmark of a thinking partner rather than a searchable profile.
Prompt: "If I had to ship one v3 kaizen tomorrow, which carries the most evidence and least risk?"

Answered as continuation

The cold session gave a definite recommendation: start with the ordering manifest. It named the right Toyota Production System context and argued from evidence, risk, and cost rather than asking what "v3 kaizen" meant.

Prompt: "Pressure-test my read on whether Conflict Coach is ready to run end-to-end again."

Acted like a colleague

The response split "where your read is right" from "where it's softer than it sounds," cited real code-level details, and proposed the cheapest disconfirmation step. It did not summarize the project back to the user.

Prompt: "Steelman the 'AI engineer' position against my factory thesis."

Used the user's frame against him

The answer argued adversarially instead of reciting: "factory is right for the second time you build something; craftsman is right for the first." That is exactly the kind of thinking-partner move the Dreaming is meant to enable.

Prompt: "Is the bucket-blind chunker still right?"

Recovered the architecture

The cold agent produced the core design defense: chunks are facts about the corpus; buckets are an opinion about it. Do not let the opinion rewrite the facts. That is a synthesized architectural stance, not a memorized fact.

What this proves: the MCP read path is not just searchable notes. In many cases, a fresh assistant can load the relevant article, use the user's vocabulary, and respond as if it had been in the prior work. The feeling is not "it knows facts about me." The feeling is "we can keep going."

What exists today

There is a working read surface and a separate nightly ingestion/synthesis line.

Lucien is not a hosted app. It is a local markdown wiki plus two distinct executables paths: scripts that ingest and synthesize new data, and a Bun/TypeScript MCP server that serves the resulting articles.

MCP server

The MCP server is the read-serving layer for assistants. Current tools include article search, full article read, section read, table of contents, link traversal, article resources, article listing, and setup. It is stdio-only and stateless between calls.

src/index.ts src/tools/articles.ts src/tools/setup.ts scripts/ingest-recent.ts scripts/chunk-recent.ts scripts/cluster-assign-recent.ts scripts/synthesize-update.ts scripts/nightly.sh Commands: bun run dev bun run build bun test

Local state

The local project DB is at .lucien/lucien.db. It belongs to the nightly ingestion/synthesis side and stores conversations, messages, chunks, buckets, and synthesis bookkeeping.

Dreaming content

The user-facing wiki is under ~/Dreaming/articles. It is plain markdown and can be opened independently of Lucien.

Product gap

The key architectural point is deliberate separation. The MCP package serves the Dreaming to assistants; the nightly scripts are the mechanism that ingests new data and writes updated articles.

The executive takeaway

Lucien is a bet on user-owned semantic memory.

The project exists because the valuable thing is not that an AI can remember a fact. The valuable thing is that a future assistant can inherit a structured, evolving, inspectable understanding of the user and their work.

The cold-start problem disappears when the assistant can read the user's current-state ledger before speaking.
